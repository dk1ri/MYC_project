Some explanation

 

A device can have rules to inform about additional conditions, which exist between the commands.

The rules will show functionality of a command, if other commands have specific status.

Limitation within a command should be described within the announcement of that command.

If status and command are in the same device, the device will deliver the rule. Otherwise a special ruledevice (systemdevice) will do this.

The rule will define, whether one or more commands are active or not, or that the status of a command  will change to another status. A application for this is to have errormessages on these conditions without the need to send these error messages as info.

Those rules lines start with “R” and are forwarded to commandsending devices.

“Q” rules are produced by the rulesdevice only send sent to the commandrouter only and are not forwarded. Up to now they are used for usermanagement. The commandrouter block commands for a interface, if necessary, for a specific user who logged in.

 

The syntax of the rules should be very simple, Most probably the rules will be generated automatic, so the focus should be the effectiveness of handling the rules by a computer. Up to now rules lines are readable, because they must be easily programmed as a part of the announcements.

 

The $<c> syntax is used to denote a command-token <c>  So the command translation of the command-router can be done easily.

 

All “R” rules are always handled by the logic device in the sequence it get the rules.

 

Rules lines can be empty to reserve the lines for later use.

 

It is not decided, whether programs and user interface must interpret the rules as well.

 

Definitions

 

<c>   commandtoken

<n> :  numeric

<s>  :  String all characters, starting with the length

<p>   parameter , sequence number in announcement

Separator:   one or more spaces or tabs

compare operator = < >

logic operator  AND OR

left side operators: ! =

right side operators: ! = < > <= >= AND OR ( )

Rules syntax

 

<left side> IF|UNLESS <right side>

 

 

left side:

command is not working (!) or automatically set

 

[!]$<c>[&<p>]| $<c>[&<p>]=<n> [!]$<c>[&<p>| $<c>[&<p>]=<n>] ...

Bracketing: !($<c>...<c>) allowed

 

right side:

left side of the compare operator is a command ($) and optional parameter (&)

right side is the status which a answer command would get. It may be more than one parameters

 

[!]$<c>[&<p>] <logic operator> <n>,[n] ... |

$<c>[&<p>] <compare operator> <n> [TO <n>] [<logic operator> $<c> [&<p>]<compare operator> <n> [TO <n>]...

 

All words / tokens separated by a separator

 

Reserved characters / words

 

( )   Bracketing

!   right side:negation, left side: not active

IF

UNLESS

AND

OR

<

>

=   left side: operator, right side: equal

TO

.   delimiter for parameter

Examples

 

!$2 IF $5 = 0   means: Command 2 will not work if status of command 5 is 0

    command 5 may be a power switch

 

!$4 UNLESS $7 = 3500000 TO 3799999 OR 7000000 TO 7199999

    means: Command 4 will work only if state 7 is in the range of .............

    This would be used if the transceiver will transmit in amateur radio band      only

    The values are transmitted values, not values of the description

 

$252="wrong address" IF $8&3 < 16 

    set errormessage if parameter 3 of command 8 is < 16
